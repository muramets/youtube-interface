## Что нужно сделать

### 1. CI/CD — автоматическая проверка кода (ВЫСОКИЙ приоритет)

#### Что это простыми словами
Сейчас: ты (или AI-агент) пишет код, запускает тесты вручную, деплоит вручную. Если забыл запустить тесты — баг попадает в прод.

С CI/CD: каждый раз, когда код пушится на GitHub, **автоматически** запускаются lint + typecheck + тесты. Если что-то красное — код не попадает в прод. Это как охранник на входе, который проверяет пропуск.

#### Что нужно сделать
1. **Агент** создаёт файл `.github/workflows/ci.yml` — это "рецепт" для GitHub, описывающий какие проверки запускать. Буквально текстовый файл с инструкциями: "установи зависимости, запусти lint, запусти тесты".
2. **Ты** настраиваешь GitHub Secrets — это переменные окружения (API ключи), которые GitHub хранит зашифрованными. Делается через GitHub UI:
   - Открываешь репозиторий на GitHub → Settings → Secrets and variables → Actions
   - Нажимаешь "New repository secret"
   - Добавляешь `GEMINI_API_KEY` и `ANTHROPIC_API_KEY` (те же ключи, что уже используешь)
   - Всё. GitHub будет передавать их в тесты автоматически.

#### Effort
- Агент: 5-10 минут (написать YAML файл)
- Ты: 5 минут (вставить 2 ключа в GitHub UI)

### 2. Sentry — мониторинг ошибок в проде (ВЫСОКИЙ приоритет)

#### Что это простыми словами
Сейчас: если у пользователя что-то сломалось — ты об этом не узнаешь, пока он не напишет в поддержку (или просто уйдёт).

С Sentry: каждая ошибка в браузере пользователя **автоматически** отправляется в Sentry dashboard. Ты видишь: какая ошибка, у скольких пользователей, на какой странице, с каким браузером. Как камера видеонаблюдения для багов.

#### Что нужно сделать
1. **Ты** создаёшь бесплатный аккаунт на sentry.io и создаёшь проект (кнопки в UI)
2. **Агент** добавляет Sentry SDK в код — `npm install @sentry/react`, инициализация в `main.tsx`, обёртка ErrorBoundary

#### Effort
- Ты: 5 минут (регистрация + создание проекта, копирование DSN ключа)
- Агент: 10-15 минут (установка, конфигурация, интеграция с ErrorBoundary)

### 3. CLAUDE.md дополнения — ускорение agentic-разработки (ВЫСОКИЙ приоритет)

#### Что это простыми словами
CLAUDE.md — это "инструкция для пилота". Сейчас она покрывает ~65% того, что AI-агент должен знать. Остальные 35% агент каждый раз переспрашивает у тебя или угадывает.

#### Что конкретно не хватает

**a) SETUP.md** — как развернуть проект с нуля.
Сейчас: новый агент (или новая сессия) не знает, какие секреты нужны, как запустить эмулятор.
После: агент читает SETUP.md и может работать автономно.

**b) PATTERNS.md** — как устроены ключевые паттерны.
Неявные знания, которые сейчас живут только "в голове" кодовой базы:
- **Sliced stores** — когда Zustand store сложный, его разбивают на "слайсы" (отдельные файлы). Как? Какая конвенция? Не описано.
- **Context bridges** — как страница (Home, Canvas, Trends) передаёт выбранные элементы в AI-чат. Паттерн существует, но не документирован.
- **Tool definitions** — как добавить новый AI-инструмент (tool) в систему. Агент должен угадывать по аналогии.

**c) Feature-to-Doc mapping** — какие фичи имеют документацию, а какие нет.
Сейчас: ~50% фич (Canvas, Video, Music, Editing) **вообще не документированы**. Агент не знает, есть ли док, и идёт искать.

#### Effort
- Агент: 30-45 минут (формализация уже известных знаний)
- Ты: 5 минут (проверить, что бизнес-intent описан верно)

### 4. Feature Gate Infrastructure — фундамент для подписки (СРЕДНИЙ приоритет)

#### Что это простыми словами
Через 2 месяца тебе нужен paywall: часть функций бесплатная, часть — по подписке. Если начать думать об этом сейчас, каждая новая фича **сразу** будет готова к paywall. Если не думать — потом придётся проходить по всему приложению и вставлять проверки.

#### Что нужно сделать
1. Добавить поле `subscriptionTier` в Firestore user document ("free" | "pro")
2. Создать хук `useFeatureGate('featureName')` — возвращает `{ allowed: true/false, tier: 'pro' }`
3. Создать UI-компонент `<ProBadge />` — иконка замка + "Upgrade to Pro"
4. В каждом feature doc указывать: эта фича Free или Pro

Пока не нужен Stripe, не нужна платёжная система. Нужен только **фундамент** — "заглушка", которая потом подключится к реальной подписке.

#### Effort
- Агент: 20-30 минут
- Ты: решить, какие фичи Free, какие Pro (продуктовое решение)

### 5. Недостающие Feature Docs (СРЕДНИЙ приоритет)

#### Что это простыми словами
По правилам проекта, каждая фича должна иметь документ в `docs/features/`. Сейчас задокументированы: Chat, Render, Trends, Video Details (Traffic). **Не задокументированы**: Canvas, Video, Music, Editing, Settings.

Без дока агент не понимает **зачем** фича существует и **как** она должна работать. Он может "починить" баг так, что сломает задуманное поведение.

#### Effort
- Агент: 1-1.5 часа (читает код, пишет доки)
- Ты: 15-20 минут (проверить business intent в каждом доке)

### 6. Централизованный Error Handling (СРЕДНИЙ приоритет)

#### Что это простыми словами
Сейчас каждый сервис обрабатывает ошибки по-своему: один кидает exception, другой возвращает null, третий ретраит. Это как если бы в одном отделе компании при проблеме звонили директору, в другом — писали email, в третьем — молча повторяли. Нужен единый протокол.

#### Что нужно сделать
- Создать общий `RetryService` с exponential backoff
- Единая классификация ошибок: transient (повторить), auth (перелогин), validation (показать юзеру), fatal (Sentry)
- Обновить ~15 сервисов на единый паттерн

#### Effort
- Агент: 40-60 минут
- Ты: ничего (техническая задача)

---

### 7. Accessibility (A11y) аудит (НИЗКИЙ приоритет, но нужен до launch)

#### Что это простыми словами
Accessibility — это возможность пользоваться приложением людям с ограниченными возможностями (незрячие используют screen reader, люди с нарушениями моторики используют только клавиатуру). Для SaaS-продукта это не просто "хорошо бы" — enterprise-клиенты часто **требуют** WCAG 2.1 AA compliance.

Сейчас: ARIA-атрибуты есть в ~20 файлах (частично), но полного покрытия нет.

#### Effort
- Агент: 2-3 часа (аудит + исправления)
- Ты: 30 минут (визуальная проверка в браузере)
